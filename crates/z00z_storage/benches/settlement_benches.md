# Live HJMT Settlement Bench Evidence

This file is the tracked evidence home for the live HJMT benchmark
harness. All measured artifacts live under `crates/z00z_storage/outputs/settlement`.
Every report below was generated by an executed run through
`crates/z00z_storage/scripts/run_storage_settlement_bench.py`; the helper now
persists Criterion JSON summaries, `/usr/bin/time -v` resource metrics, inlined
note files, and Stage 13 scenario JSON summaries from the real measured path.
Measured numbers remain evidence only and must never become protocol constants
or proof-visible authority.

Phase 058 archive-home honesty is explicit here: this file plus
`crates/z00z_storage/outputs/settlement/` are the only live measured bench
evidence homes. Any older alternate archive-home wording is a retired planning
alias, not a second archive path and not a missing live home.

`cargo bench ... -- --help` is not an accepted provenance path for this phase.
Bench owners now gate side outputs behind `should_emit_side_outputs()`, so help
output no longer refreshes note or meta files.

## 🔒 Provenance Rules

- Every measured report above comes from a non-`--help` execution path and records the exact command used for that run.
- `settlement_proofs_batch.md` is a filtered execution of the same live `settlement_proofs.rs` bench home; it exists so Phase 055 can refresh the additive batch-proof surface without introducing a parallel bench crate or rerunning unrelated legacy proof lanes.
- The filtered batch report rewrites `settlement_proof_sizes.md` in `batch_only` scope, so it does not recalculate unrelated proof-size bytes and does not duplicate verify timing already owned by `hjmt_batch_verify`.
- The canonical `hjmt_batch_` batch-only path now refreshes only the light live prove/bytes matrix for `count in {2,8,32}` directly in note scope, does not rerun the full `hjmt_batch_proof_bytes` Criterion matrix inside `settlement_proofs_batch.md`, and leaves `128/1000/1024` to the full `settlement_proofs.md` benchmark and stress lanes; repeated-loop runtime authority remains with `hjmt_batch_verify`.
- Batch-only note files now record the actual filtered bench command and lane selector from the run, so narrow `hjmt_batch_proof_bytes/...` probes stay explicitly partial instead of claiming full `hjmt_batch_` scope.
- Each non-`--no-run` storage bench report now archives a sibling `*.timing.tsv` trace and inlines an `Internal Stage Timing Slices` section derived from it. Mutation-facing HJMT reports must preserve separate `hjmt_plan_ops`, `hjmt_child_commit`, `hjmt_parent_commit`, and `hjmt_journal_sync` slices instead of one merged commit number.
- `hjmt_batch_proof_bytes` now measures cold prove plus raw encode per iteration on the same synthetic fixture. It is not a cache benchmark, not a live-update benchmark, and not a compression benchmark.
- `prove_time_us` and any note-level `verify_time_us` column in `settlement_proof_sizes.md` are one-shot wall-clock samples from the note path only; Criterion timing remains the authority for repeated-loop runtime, while cache semantics stay with `proof_generate_cache_state` and `proof_verify_cache_state`.
- Filtered debug runs that do not include `hjmt_batch_proof_bytes` now set `proof_note_scope=skip`, so partial probes cannot overwrite `settlement_proof_sizes.md` with note content that lacks a real batch-note source.
- `scenario_1_hjmt_workload.md` intentionally contains no Criterion results; it captures `/usr/bin/time -v` metrics for the exact release command `cargo run --release -p z00z_simulator --bin scenario_1`, verifies that each Stage 13 JSON artifact was refreshed during the current run, and then inlines those summaries. Debug-only release-feature rejection remains a separate guard owned by `bash scripts/audit/audit_release_feature_guards.sh`.
- The HJMT diagnostic note and the proof-size, reload, and recovery notes are now consumed only from executed benchmark runs and are inlined into the generated reports.
- Fixed-bucket and adaptive HJMT measurements are both present; adaptive metrics remain advisory until the full review loop is clean.
- Phase 27 selects `Closed by Stage13 evidence`: there is no standalone measurement sidecar file, and the canonical local proof-size guardrail remains the verified Stage 13 proof-size/cache/replay/tamper artifact packet plus `crates/z00z_simulator/tests/scenario_1/test_scenario1_stage_surface.rs`.
- Any future default-switch decision must cite the measured report files above, not the values copied into this summary.
