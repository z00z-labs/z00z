8. **S1-01 authority-reviewed theorem amendment (2026-07-17):** the measured
   fixed-depth wide-Poseidon root/frontier path is removed. Fresh bounded setup
   exhausted the 24 GiB emergency address-space ceiling after fixed empty-tree
   constants and lossless frontier compression, while the root was never an
   independent public or authority endpoint. The audited replacement retains
   the exact source-byte theorem without an inner proof, second accumulator, or
   digest-only equality.
   The sole expander obtains one stack-local
   `RecursiveTraceCanonicalChunkV2`, emits one derived `SourceMemoryWrite`
   immediately before the matching `TraceChunk`, and never materializes an
   additional record/chunk vector or source spool. The circuit enforces this
   induction:
   - a writer is legal only with an empty pending window and an active,
     non-EOF source context;
   - its source ordinal equals the active source ordinal and its chunk ordinal
     equals the exact next source cursor;
   - a pending writer forces the immediately following opcode to be
     `TraceChunk`;
   - the reader equality-constrains source ordinal, chunk ordinal, chunk count,
     byte count, all 64 bytes, and the zero tail to the pending window, then
     clears the window;
   - that same reader is appended byte-for-byte to both the source-record and
     global-trace SHA-256 contexts, with constrained canonical header, repeated
     chunk count, non-final width, exact final length, EOF, padding, chaining,
     and final authority-bound digest.
   Therefore, by induction over the source cursor, every consumed byte has one
   ordered writer and one equal reader. Any different byte stream either fails
   one of those R1CS equalities/schedule gates or requires a collision in the
   already declared SHA-256 assumption A-03. The removed Poseidon root/frontier
   added no separate binding lemma because all prior bytes were already folded
   into the constrained SHA chaining states and the root was not compared to an
   independent endpoint. This is a theorem-preserving resource amendment, not
   a native assertion, reduced-cap relation, or digest substitute for direct
   equality. A bounded payload arena, second source/circuit owner, runtime
   selector, source rewrite, and mutable memory API remain forbidden. The
   derived `SourceMemoryWrite` is not a source record or SHA input and cannot
   appear without its matching `TraceChunk`.

### Client-centric notary intent

The downstream architecture MUST treat Z00Z as a notary of current ownership,
not as a permanent public copy of every user's transaction history:

1. Canonical finality remains quorum/certificate based at checkpoint cadence.
   Recursive proving consumes already-finalized checkpoints asynchronously and
   MUST NOT delay, revoke, replace, or authorize canonical finality.
2. The network MUST retain enough current sharded HJMT/state chunks to serve a
   fresh membership or non-membership witness. A root without retrievable current
   leaves/chunks is not a usable client-centric state service. Sharding is a
   launch requirement, not a later scale-out option: the first active profile
   MUST use the existing deterministic `ShardRouteTable` / `ShardPlacementTable`
   / distributed-HJMT owners with more than one logical shard, per-shard replicas

   shard-local execution, a generation-bound route table, and copy/verify/
   delta-catch-up/CAS resharding; it does not authorize a parallel state theorem.
3. A wallet MUST verify `network trust anchor -> committee keyset/threshold/
   rotation manifest -> finality certificate -> authenticated checkpoint root ->
   transaction/owned-output inclusion -> HJMT membership -> expected owned
   output`. Minimal historical committee manifests remain content-addressed and
   retrievable; a wallet MUST NOT fetch PP, PK, or the Nova VK for its default
   security path.

   window. Permanent public receipt history is not required for recovery or
   spending; complete history comes only from wallet-owned backup.
5. Raw transaction packages, exact transaction-proof bytes, and the minimum

## 🔑 Security Verdict And Threat Model

`crypto-architect` verdict: **Risky but salvageable; execution-ready only with
the gates in this plan.** The corrected theorem has no unresolved structural
S0, but the following S1 integration failures remain possible until their live
code, negative-test, and trace gates pass:

- a proof-selected or unpinned parameter/verifier bundle;
- successful backend verification without exact all-limb `z_N` comparison;
- per-block Nova restart or first-step off-by-one;
- unbounded proof/trace/spool decoding before a cap check;
- source/feature drift that replaces Nova's production `OsRng` sampling with
  deterministic, repeated, caller-controlled, or test-only entropy;
- authority/config/key rotation between decode, verification, receipt, write,
  and reload, producing one mixed-generation acceptance;

## 🚨 Mandatory Three-Model Falsification Gate

Every suspected theorem, cryptographic, or verifier defect and every claimed
fix MUST be checked in this order:

1. **Model A — logical attack:** construct two different semantic executions,
   witnesses, or endpoints satisfying the same bound statement, or identify the
   exact invalid reduction premise. Retain algebra/bytes and the violated lemma.
2. **Model B — differential:** run identical typed trace bytes through the
   independent evaluator, StepCircuit, native transcript, in-circuit transcript,
   and independent test-only reference. Compare every field, absorb, challenge,
   counter, root, `z_0`, and `z_N`.
3. **Model C — unchanged-verifier exploit attempt:** repeatedly try to make the
   authority-pinned unmodified Nova 0.73.0 compressed verifier and exact project
   endpoint accept proof bytes for a false transition. Preserve unrelated
   encoding/checksum fields so the candidate reaches the real verifier, compare
   every returned limb, repeat across clean processes, and retain minimized
   fixtures and commands.

A decoder, checksum, shape verifier, native precheck, mock, reduced circuit,
receipt parser, or rejection before the real compressed verifier is not Model
C evidence. A candidate is closed only when the real verifier plus exact
endpoint comparison rejects it, or when a valid proof of non-applicability is
mapped to code and regression tests.

Only a candidate passing A, B, and C may be labelled a critical backend
vulnerability. It then requires three independent reproductions and a retained
exploit before a local fork is considered. No current candidate passed all
three models; therefore this plan MUST NOT vendor/fork Nova or Poseidon.

## 🚦 Mandatory Live Acceptance Closure Gate

The implementation MUST expose one production checkpoint-evidence ingress in
the existing recursive-proofs adapter and prove this exact typed sequence:

```text
authority snapshot → object-family/cap → bounded strict decode → canonical
re-encode/curve validity → bundle identity → unchanged Nova verify → complete
z_N comparison → statement/link/predecessor → storage endpoint reload → atomic
shadow-sidecar write → exact byte reload → repeat Nova/endpoint checks → receipt
```

Each arrow consumes a private linear typestate value. No stage implements
`Clone`, `Copy`, `Serialize`, `Deserialize`, or `Default`; no public constructor,
boolean, enum verdict, persisted sidecar, receipt bytes, log record, or test
fixture can recreate it. The authority generation/digest, bundle digest, proof
digest, statement/link/predecessor, cumulative step count, complete `z_0` and
`z_N`, storage generation, and checkpoint ID remain equal at every stage.
Rotation, reorg, cancellation, timeout, crash, or any mismatch invalidates the
attempt and requires a fresh start before decode.

The same transitions emit a bounded, private `VerificationGateTraceV2`-style
record from their actual branches. It records gate ID, order, authority
generation, public object digests/counters, and typed terminal reason only. It
never records witness material, private IVC state, raw proof bytes, paths,
secrets, or unrestricted backend errors. The trace is evidence and cannot make
verification succeed. Tests fail if any receipt-producing edge is missing,
duplicated, reordered, bypassed, or absent from the theorem-to-code matrix.
Storage may persist an untrusted non-authoritative sidecar, but neither its
`Accepted` verdict nor a deserialized receipt can authorize canonical admission
or skip fresh backend verification.
