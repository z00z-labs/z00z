# 069-09 Verified Receipt Storage Integration

<objective>
Bind actual Nova and Plonky3 verification receipts into the existing storage-
owned sidecar, epoch manifest, archive, PQ-anchor, snapshot, pruning, validator,
and watcher seams through a rollup-owned dependency-safe integration API,
without creating replacement schemas, reversing crate dependencies, or changing
canonical checkpoint admission. Migrate retention to a client-centric bounded
challenge ring plus compact permanent epoch anchors, and add the finalized
wallet-receipt/inclusion boundary, current-state seed recovery, and a fail-closed
registry/config/outbox handoff for the encrypted offline-recipient delivery
mailbox implemented in Phase 071. Phase 069 MUST NOT add mailbox codecs, stores,
providers, activation, ACK/GC, benchmarks, or delivery scenarios. Make current-state
sharding and RS challenge-archive sharding mandatory in the first active
profile; do not preserve the former one-shard/full-replica MVP escape hatch.
</objective>

## 🔑 Client Retention Wallet Contract

### Data classes and ownership

1. `transient_prover`: decoded witnesses, assignments, trace chunks, sort runs,
   temporary proof buffers, and derived copies. These MUST be bounded and
   zeroized/dropped on every exit; they MUST NOT enter archive manifests.
2. `active_epoch`: Nova accumulator/recovery state, Plonky3 base/intermediate
   proofs, open manifests, and staged evidence. These MUST be bounded by Plan 06
   and Plan 10 pending-work limits and become delete-eligible after verified
   epoch/history commit plus recovery grace.
3. `challenge_window`: canonical raw transaction packages, each unique exact
   transaction-proof blob stored once by digest, and only the replay/delta/
   witness material that cannot be deterministically reconstructed, plus
   non-epoch per-block header/QC bodies needed for in-window audit. The default
   window is exactly `1,555,200` finalized blocks from
   `da_publication_ready`; expiry MUST be height-based and fork-bound.
   For DA-ready finalized height `h`, checked arithmetic sets
   `expiry_height = h + 1,555,200`; bytes remain in `[h, expiry_height)` and
   become time-eligible only at finalized canonical height `>= expiry_height`.
4. `permanent_compact`: genesis/context commitments, compact immutable
   `EpochCloseAnchorV2` / `EpochEvidenceAnchorV2` records, MMR/history peaks,
   the epoch-close finality certificate that authenticates each close anchor,
   the minimal content-addressed committee public-key/threshold manifests and
   rotation certificates needed to verify historical finality certificates,
   parameter rotation commitments, and deletion-ticket audit digests.
   Raw packages, non-epoch header/QC bodies, state chunks, snapshot bodies, full
   proof bodies, user receipts, and derivable witnesses MUST NOT enter this class.
5. `live_state_and_recovery`: current canonical head, current sharded HJMT/state,
   and latest three verified COW snapshot heads plus referenced live chunks.
   This class is required for ownership witnesses/bootstrap and rolls forward;
   it MUST be capacity-accounted separately from permanent historical growth and
   MUST NOT be deleted merely because a challenge window expires.

### Nova proof-body lifecycle

Plan 09 is the sole semantic owner of compressed Nova proof-body rotation,
retention, disposition, and deletion predicates. Plan 06 supplies verified
immutable body/chain-root/generation/reference facts; Plan 10 consumes a committed
ticket/state generation to perform crash-safe physical GC. Neither may infer or
alter this policy.

| State | Entry gate | Legal successor | Body requirement |
| --- | --- | --- | --- |
| `OpenEpoch` | Active Plan 06 fold lineage and exact generation are current. | `ClosedAwaitingPq` or body-less terminal `GapRecorded` under cap backpressure. | Running accumulator/recovery snapshots are local Plan 06 artifacts, not this proof body. |
| `GapRecorded` | Compression was not admitted because the finite pending-PQ/body cap or safe-disable gate was already active; chain root and permanent `not_promotable` reason are committed and `body_digest` is absent. | none | No body exists, so no deletion ticket or physical GC is legal; it is never successful/promotable evidence. |
| `ClosedAwaitingPq` | Epoch/range closed; accepted scheduled body, if produced, verified and committed with Nova chain root/body digest. | `PqSuperseded` or `Abandoned`. | At most two accepted bodies per epoch; retry/temp files never count or persist. |
| `PqSuperseded` | Independent exact-range Plonky3 verification, evidence join, and later certified evidence-MMR inclusion all bind the same epoch/generation. | `DeleteEligible`. | Body stays until grace/reference/hold gates and ticket CAS pass. |
| `Abandoned` | An existing body's terminal job hit the pending-PQ count/byte cap or explicit safe backend-disable; no hold/live reference remains and a body-digest-bound permanent `not_promotable` disposition is committed. | `DeleteEligible`. | `body_digest` is mandatory; never successful/promotable Nova evidence; the body remains until its scoped ticket. |
| `DeleteEligible` | Exact scope-specific `RetentionTicketV2` and retention-head CAS committed. | `Deleted`. | Readers return typed retired/abandoned status; body is no longer required for visibility. |
| `Deleted` | Exact content-addressed body unlinked and ledger reconciled. | none | Compact chain/body/disposition/ticket digests remain; physical leftover cleanup is idempotent. |

The default profile MUST enforce `max_retained_proof_bodies_per_epoch = 2`,
`max_pending_pq_epochs = 8`, `max_nova_retained_proof_bodies = 16`,
`max_nova_retained_proof_body_bytes = 2,097,152`, and
`post_pq_grace_certified_epochs = 2` with checked multiplication. The first body
is the scheduled epoch body; the second is an explicitly authorized local on-
demand body. There is no retained retry allowance. The legacy-named 128-MiB
`max_epoch_nova_archive_bytes` is only a non-default per-block compression
benchmark/fuzz aggregate cap and MUST NOT be read as retention capacity or enable
per-block persistence/publication.

Every accepted `NovaProofEnvelopeV2` is one canonical content-addressed object.
`RecursiveCheckpointSidecarV2`, `EpochManifestV2`, evidence anchors, and the
retention ledger MUST store only bounded typed references/digests and MUST NOT
duplicate `proof_bytes`. Body counts/bytes and tickets apply to unique envelope
objects; compact reference, verifier-receipt summary, disposition, and ticket
digests remain after the envelope is retired.

On the ninth pending epoch or earlier byte-cap exhaustion, the system MUST stop
new Nova compression/publication first and emit `NovaEvidenceBackpressure`; it
MUST continue canonical finality and the running fold/hot recovery path within
its independent measured cap. New epochs enter terminal `GapRecorded` with no
body/ticket/GC rather than fabricated evidence. This does not release the separate 90-day challenge
ring: if Plonky3/history remains absent, challenge deletion remains blocked and
the operator enters stop/split or capacity action.

The normal ticket scope `nova_pq_superseded_body` requires the exact Plonky3/
join/evidence-MMR coverage, two later certified epochs, no hold/dispute/fork/live
reader/re-prove/job/reference, matching generation/VK/manifest, retained compact
digests, and ledger CAS. The degraded scope
`nova_unpromoted_abandoned_body` requires the permanent `not_promotable` gap,
terminal shadow job, no hold/live reference, exact body namespace/digest, and CAS;
it does not claim Plonky3 or Nova success. Both scopes MUST reject targets outside
the Nova proof-body namespace and MUST NOT address PP/PK/VK, recovery snapshots,
raw packages, transaction proofs, witnesses/deltas, current state, or anchors.

Deletion ordering is irreversible and fail-closed: validate ticket/references;
durably commit ticket + disposition/tombstone + new retention/evidence head by
CAS and fsync; then unlink the exact content-addressed body; then reconcile
`Deleted`. Before head CAS, a missing body is corruption. After head CAS, readers
return `NovaBodyRetired` or `NovaBodyAbandoned`; a remaining body is unreachable
garbage removed idempotently. Directory enumeration, wall clock, filename age,
proof digest alone, or current config never authorizes state or deletion.

`NovaAccumulatorSnapshotV2` is explicitly outside this state machine. Plan 06
keeps the active accumulator plus two verified recovery snapshots, while Plan 10
journals local snapshot retirement only after a newer head is durable and rollback/
reader references clear. Production activation remains blocked while
`max_nova_hot_recovery_bytes = 0`; Plan 11 must replace it in a new measured,
authority-pinned profile generation with a positive finite cap.

### Pruning ticket

`RetentionTicketV2` MUST bind one typed scope, target namespace/range/object
digests, exact predicate bitmap, predecessor disposition/head, and CAS generation.
For `expired_challenge_objects` it proves: window expiry on the current finalized
chain; no dispute, legal/operator hold, or unresolved fork; final epoch/history
verification receipts; evidence-anchor inclusion in a later quorum-certified
close; complete compact anchor/MMR lineage; a verified newer bootstrap snapshot;
the configured RS reconstruction/placement threshold and unpredictable retrieval
audit; no live reference to an old VK/parameter generation; and retention-ledger
CAS. The two Nova scopes prove only their separate lifecycle predicates above and
can address only Nova proof bodies. A ticket grants idempotent deletion of exactly
its declared object set. It MUST NOT authorize current-state, permanent-anchor,
active-challenge, recovery-snapshot, key-material, cross-namespace, or
unreferenced-path deletion.

Deletion is an irreversible loss of protocol-guaranteed old-byte replay, not a
cryptographic proof that no optional copy exists or that the proof system can
never fail. Independent users/community archives MAY retain expired objects,
but consensus MUST NOT rely on them. Phase 069 MUST implement and exercise the
`retention_ticket_v2` stage in local/test profiles; a production/operator default
MUST remain `declared_only` until an explicit retention promotion approves the
Plonky3/history integration audit, emergency stop/rollback, capacity alarms, and
restore/re-prove drill. Storage savings MUST NOT be marketed before that gate.
After expiry, bootstrap is from a certified recent snapshot plus permanent
epoch-close certificates, committee rotations/manifests, anchor/MMR lineage,
and optional verified history evidence—not a genesis-to-tip raw replay. Any
operator requiring indefinite raw re-execution MUST opt into an archive before
expiry; the base protocol MUST NOT imply that capability after deletion.
