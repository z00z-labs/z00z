# 069-09 Verified Receipt Storage Integration

<objective>
Bind actual Nova and Plonky3 verification receipts into the existing storage-
owned sidecar, epoch manifest, archive, PQ-anchor, snapshot, pruning, validator,
and watcher seams through a rollup-owned dependency-safe integration API,
without creating replacement schemas, reversing crate dependencies, or changing
canonical checkpoint admission.
</objective>

  through `069-08` artifacts.

## inputs

- Actual Nova/Plonky3 receipts, authority-pinned V2 verifier bundles (never
  prover material), existing inherited schemas/stores, new epoch schemas,
  validated config/path/cap/lifecycle policy.

## outputs

- Strict storage-owned receipt envelope/reference and immutable bindings from real receipts into sidecar/manifest/anchor artifacts.
- Rollup-owned `recursive_evidence` integration seam that depends on both
  `z00z_recursive_proofs` and `z00z_storage`, while neither dependency points
  back to the rollup or to each other in the forbidden direction.
- Archive/retrieval/snapshot/pruning and validator/watcher non-authority regression/integration evidence.
- Retention/publication policy: verifier roles cache a finite, authority-pinned
  VK bundle once per generation; PP/PK never enter validator/watcher/wallet or
  proof-envelope wire paths. Wallet finality uses finality certificate, committed
  root, and HJMT membership proof rather than recursive proof verification.
- Rotation/archive/deletion policy: retain PP/PK only in the encrypted prover
  recovery store for the configured active-generation rollback/in-flight window,
  then securely delete; retain each content-addressed VK manifest/digest and
  retrievable VK until no retained proof references it; retain Nova sidecars
  through their Plonky3 epoch window; permit local pruning only after completed
  V2 epoch proof/manifest plus inherited archive-replication and retrieval-audit
  gates; retain final epoch proof/manifest and rotation ledger as archive evidence;
  never treat DA as that archive or allow an archive node to prune it.

## dependencies

- `069-06` real Nova chain and `069-08` real recursive epoch proof.

## acceptance_tests

- Only an actual backend receipt can populate verified evidence.
- Receipt success is represented only by the backend crate's private-constructor
  `VerifiedProofCapabilityV2`; storage persists a backend-neutral opaque evidence
  record and never interprets decode/shape/digest success as verifier success.
- Canonical artifact bytes, replay bytes, link continuity, and `CheckpointProofSystem` remain unchanged.
- Epoch/archive/PQ/snapshot/pruning bindings and effective size caps fail closed.
- Validator/watcher/archive/snapshot evidence remains non-authoritative.

## simulation_gate

- Local filesystem/provider boundaries may use deterministic adapters; proof verification, storage codecs/writes, archive receipts/audits, snapshot/pruning rules, validator and watcher checks are real project primitives.

## negative_tests

- Missing/forged/drifted receipt or opaque record, direct caller construction,
  storage-write bypass, shape-only success, arbitrary bytes, wrong
  statement/link/archive/Nova/epoch/context/parameter digests, cap overflow,
  unpinned IPFS, non-independent replicas, missing/stale audit, incomplete
  snapshot, early/archive-node pruning, snapshot-as-trust-root, sidecar
  authority, `VERIFIED` admission, and legacy PQ bytes presented as a signature
  or verifier result.
