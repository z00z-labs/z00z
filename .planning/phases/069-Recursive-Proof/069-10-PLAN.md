# 069-10 Bounded Prover Lifecycle And Atomic Recovery

<objective>
Add bounded, cancellable, observable proving and wallet-backup lifecycles plus
three explicit crash-consistent commit protocols: canonical checkpoint head,
recursive multi-artifact evidence, and wallet local-state/remote-backup outbox.
Lag, timeout, memory/queue exhaustion, restart, reorg, parameter rotation, or
backup-provider failure must affect only their owning state machine; stale or
partial state never appears complete and a finalized payment is never rolled
back by proof or backup failure.
</objective>

## three_state_machine_atomicity_contract

There is no safe global transaction across consensus, DA, recursive workers,
wallet storage, and a remote backup provider. The implementation MUST use
digest-bound idempotent sagas with exactly three top-level logical transaction
domains. Each domain has one authority-local durable head/commit boundary;
per-shard promotion and remote-provider operations are subordinate idempotent
outbox consumers, not extra authorities or one distributed transaction:

1. **Canonical checkpoint:** `Staged -> DaReady -> QuorumCertified -> Committed`.
   Immutable content-addressed blobs may be written before commit. External DA
   readiness is obtained idempotently, and the quorum certificate MUST bind the
   exact checkpoint/header, state root, active route generation/table digest,
   touched shard set, staged shard/chunk/delta roots, per-shard prepare/quorum
   receipts, DA/publication commitment, context,
   term/committee generation, and predecessor. One immediate-durability database
   transaction then installs the new state generation, global state root,
   `CurrentStateShardManifestV2` digest, canonical head, certificate/reference,
   and durable shard-promote/recursive/archive plus generic finalized-output
   handoff outbox events. Replica-local
   promote transactions consume the exact event idempotently; no cross-node DB
   transaction is claimed. Recursive state is excluded.
   A crash after QC but before commit replays the same certificate/commit; a
   conflicting certificate or head CAS fails closed.
2. **Recursive evidence:** proof work follows `Queued -> Proving -> Proved ->
   ProofVerified`, while the independent archive lane follows
   `AvailabilityPending -> AvailabilityVerified`; a generation/root-bound join
   alone advances `ReadyToCommit -> EvidenceCommitted`, with
   `Superseded/Rejected` terminal branches. Immutable
   proof/receipt/sidecar/manifest/anchor objects become visible only through one
   verified commit record. Missing/failed proof creates backlog/retention hold,
   never canonical rejection. Readers MUST follow the committed evidence head,
   not directory enumeration or existence of one file.
   Nova proof-body retirement is a subordinate state machine owned semantically
   by Plan 09: `OpenEpoch -> ClosedAwaitingPq -> PqSuperseded -> DeleteEligible ->
   Deleted`, with an explicit body-bearing `ClosedAwaitingPq -> Abandoned ->
   DeleteEligible` degraded branch. Body-less `OpenEpoch -> GapRecorded` is
   terminal metadata and MUST NOT invoke Plan 10 deletion/GC. Plan 10 may execute only the exact committed transition/
   ticket generation. The durable delete order is ticket + disposition/tombstone
   + retention/evidence-head CAS first, exact body unlink second, `Deleted`
   reconciliation last. Readers before CAS require the body; readers after CAS
   receive typed `NovaBodyRetired`/`NovaBodyAbandoned`. An extra post-CAS body is
   harmless unreachable garbage; a pre-CAS missing body is corruption.
3. **Wallet and backup:** one local database transaction commits finalized
   receipt, owned/spent output changes, scan cursor, wallet-state root, and a
   backup-outbox job. Phase 069 commits no mailbox ACK or upload outbox. The
   remote backup worker executes `Pending -> Uploaded ->
   ShardsUploaded -> Reconstructed -> ReadVerified -> HeadCommitted ->
   GcEligible`. Provider failure retries the
   outbox and MUST NOT fail or roll back the payment transaction.

Phase 069 stops at this generic three-domain boundary. It MUST NOT implement a
mailbox-specific outbox row, activation consumer, upload/readback, ACK,
tombstone, or GC. The Phase-071 handoff nevertheless freezes the future rule:
mailbox upload/replica/readback/activation/ACK/tombstone/GC are subordinate
provider operations joined by immutable entry/output/activation digests and
idempotency keys; they never create a fourth top-level transaction, claim cross-
node atomicity, or delay finality. Sender local GC follows write quorum plus
authenticated readback, not recipient ACK. Recipient ACK is enqueued only in the
authoritative wallet transaction that installs receipt/output/cursor; replay
occurs after commit and GC is confined to the mailbox namespace/original expiry.
The future sender row binds final transaction digest, immutable entry bytes/
digest, exact signed `ReceiptMailboxAdmissionV1` bytes/digest, and route/profile
generation. Phase-071 recovery replays those exact persisted bytes and never
rebuilds, re-encrypts, or re-signs from mutable current config.

Consensus-critical filesystem writes MUST use
`temp -> write -> file sync_all -> rename -> parent-directory sync_all` on
supported platforms. If the filesystem cannot provide the required semantics,
authoritative visibility MUST live in the durable database commit record.
`write_file()`-style flush/rename alone or sequentially written artifacts are
not a crash-consistent logical transaction.

Every saga/idempotency key MUST bind network, object kind/version, height/range,
checkpoint ID, statement/context/predicate digest, generation, and predecessor.
Recovery scans journals/heads, verifies canonical bytes/digests, and either
completes the exact transaction or quarantines it; it MUST NOT guess from
timestamps, filenames, partial directory contents, or provider listing order.

Local `NovaAccumulatorSnapshotV2` GC is not proof-body retention. It MUST use a
separate recovery-journal entry binding old/new verified snapshot heads, lineage,
rollback floor, readers, and CAS generation. Physical removal occurs only after
the newer head and journal record are durable and the active-plus-latest-two set
remains complete. A Plan 09 proof-body ticket cannot address these snapshots.

### Wallet backup V5

- Reuse the repository-approved Argon2id and XChaCha20-Poly1305 backup
  construction; do not introduce a new cipher/KDF/MAC. A random data-encryption
  key MUST be wrapped by an approved password-derived KEK; an independently
  authenticated platform/recovery-key wrap MAY be added only under an explicit
  recipient/policy manifest. Plaintext DEKs and an unreviewed weaker alternate
  unwrap path are forbidden.
- The encrypted inner header MUST bind `backup_seq`, parent backup digest,
  device ID, checkpoint height/root, latest trusted finalized close-anchor
  digest/generation, wallet-state root, receipts root, payload digest,
  encryption generation, and schema/network context. The public outer
  header MUST expose only version, exact Argon2id algorithm/parameter profile,
  random salt, payload nonce, ordered key-wrap algorithm/generation/nonces and
  wrapped DEK bytes, ciphertext length, codec generation, and the domain-
  separated schema/network-context digest needed to reject cross-network
  substitution; it MUST NOT expose a stable wallet identifier or any additional
  network/account metadata.
- Every backup generation MUST use a fresh CSPRNG salt, fresh data-encryption
  key, one fresh 24-byte XChaCha20 payload nonce, and a distinct fresh nonce for
  every AEAD key-wrap recipient; reuse of any `(key, nonce)` pair is forbidden.
  A canonical immutable `HeaderCoreV5` binds version, algorithms, KDF profile,
  lengths/counts, schema/network context, and encryption generation and is AEAD
  associated data for every DEK wrap. The payload AEAD associated data is the
  canonical header core plus the ordered authenticated key-wrap entries. Mutable
  provider metadata/ETags are excluded; algorithm/KDF/length/generation/wrap
  insertion, removal, reordering, or substitution MUST reject.
  The decoder MUST enforce approved minimum and maximum Argon2id memory/time/
  parallelism, salt/nonce/wrapped-key counts, and ciphertext length before
  allocating KDF memory or decrypting; weak/downgraded or excessive parameters
  fail closed.
- Provider interface MUST be capability-minimal:
  `put_immutable`, `get`, `compare_and_swap_head(expected_version)`, `list`, and
  `delete`. Google Drive, IPFS, WebDAV, local filesystem, or another provider is
  an untrusted availability adapter; provider bytes/ETags are never wallet truth.
- The wallet MUST canonical-encode and AEAD-encrypt the complete logical backup
  generation before erasure coding. Only the authenticated ciphertext envelope
  is split with the initial required versioned `rs_3_5_v1` Reed-Solomon profile
  (`k=3,n=5`). Any later profile change requires a new registered backup-policy
  generation and migration; provider input never selects it. Plaintext, DEKs,
  password-derived KEKs, key-wrap plaintext, or independently meaningful wallet
  records MUST NOT be sharded or uploaded. `WalletBackupShardManifestV5` binds
  the ciphertext root/length, exact `k/n` and codec generation, ordered shard
  indices/digests/lengths, provider object IDs through opaque digests, placement/
  failure-domain status, backup generation digest, parent, and head CAS token;
  it exposes no stable wallet ID or additional account metadata.
- Upload success is insufficient. The worker MUST read back, authenticate,
  reconstruct the exact ciphertext from at least `k` distinct verified shards,
  authenticate/decrypt, canonical-decode, and recompute the wallet-state/
  receipts/payload roots before CAS of the head to the shard-manifest digest.
  The implementation MUST exercise every allowed missing-shard pattern within
  the selected durability target. Only after CAS may older logical generations
  become GC candidates.
- At least the latest three verified generations MUST remain. Concurrent-device
  parent mismatch creates a detectable backup fork: receipts may be merged by
  identity, but spend/current-state truth is resolved only from canonical chain
  state. Rollback to a lower sequence or non-ancestor head rejects.
- “Three generations” means three independently reconstructible logical
  generations, each retaining at least its bound `k` valid shards. GC MUST NOT
  delete a shard from any protected generation below `k`, delete the old head
  before the new reconstruct/decrypt/readback/CAS succeeds, or treat several
  objects in one provider account as independent failure domains. A single
  Google Drive/IPFS/WebDAV/local provider remains permitted as an untrusted
  ciphertext store, but its status MUST report one failure domain rather than a
  false multi-provider durability claim.
- Existing Backup V4 bytes are bounded read-only import input, not a V5 head or
  a rollback-safe generation. Migration MUST authenticate/decrypt and validate
  the exact V4 object locally, bind its canonical digest in a migration record,
  create a fresh V5 generation with fresh salt/DEK/nonces and an explicit
  genesis-or-migration parent marker, then complete shard upload,
  reconstruction, decrypt/readback, and head CAS. V4 bytes MUST NOT be
  relabeled, rewritten in place, or counted among the three protected V5
  generations. The last valid V4 recovery copy MUST NOT become GC-eligible
  until three independently reconstructible V5 generations and their verified
  ancestry exist.
