- First run `cargo metadata --format-version 1` and audit direct dependency
  owners before changing Cargo files.
- Add or keep direct crates only where the owning crate exercises the API:
  `redb` or the existing `z00z_storage` seam for durable consensus records;
  `proptest` for threshold/blob/restart properties; `tracing` and
  `tracing-subscriber` for asserted structured run evidence; `object_store`
  only for exercised Celestia-local retention/retrieve behavior; and
  `ed25519-dalek` only behind the project signer/verifier trait if
  `z00z_crypto` cannot satisfy the vote-signature seam.
- Treat `bytes`, `borsh` or a project-owned canonical binary codec,
  `metrics`, and `prometheus` as explicit Phase 067 dependency candidates:
  `bytes` belongs only where transport envelopes, payload chunks, blob bodies,
  or backend proposal bytes are exercised; `borsh` belongs only where signed or
  hashed canonical bytes need a stable non-JSON encoding; `metrics` and
  `prometheus` belong only where local backend/devnet metrics are emitted and
  asserted by tests.
- Keep `hotstuff_rs`, `libp2p`, `celestia-client`, `celestia-rpc`,
  `celestia-types`, `arc-malachitebft-*`, `blsful`, `reed-solomon-*`, and
  `openraft` blocked until a local adapter contract exists and tests exercise
  the crate.
- Tooling order is fixed: runnable `z00z_rollup_node` command first,
  manifest/process command tests second, optional Docker/local-task harness
  third, final release and GSD review gates last.
</dependency_installation_sequence>
